# Photo Grid Sort Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a toolbar sort control (Date taken / Date imported, newest / oldest) to the photo grid on Library, Albums, and Search — not Trash — and thread the chosen sort end-to-end through the grid listing and the photo detail view's prev/next + film-strip navigation.

**Architecture:** A single source of truth for the four sort values lives in `@lumio/shared`. The server maps a sort to a Prisma `orderBy` via a `photoOrderBy()` helper, used by the list and neighbor queries. The client persists the choice in `localStorage` (a `useGridSort` hook mirroring `useGridColumns`), renders a `GridSortMenu` dropdown (mirroring `GridViewMenu`), sends `sort` in the API query, carries it on photo detail hrefs, and remounts the grid via its React `key` so a sort change refetches cleanly from page 1.

**Tech Stack:** Next.js (App Router) + React 19, TypeScript, Prisma/PostgreSQL, Zod, Vitest, Tailwind + shadcn/ui (DropdownMenu), lucide-react icons. pnpm workspace monorepo (`@lumio/web`, `@lumio/db`, `@lumio/shared`).

## Background facts (verified in the codebase)

- Ingest sets `sortDate = takenAt ?? new Date()` (`packages/ingest/src/store.ts:35`). So **`sortDate` already means "taken date, falling back to import time"** — `taken-desc` is byte-for-byte today's default ordering, and EXIF-less photos stay chronological for free.
- The current order is the constant `PHOTO_ORDER = [{ sortDate: "desc" }, { id: "desc" }]` (`apps/web/src/lib/photo-order.ts`), used by the list queries **and** the detail-view neighbor queries.
- `usePhotoPages` fetches only on mount and resets only on remount; album/search views already remount the grid via a React `key` (`apps/web/src/components/photo-grid/use-photo-pages.ts:26,59-63`). **A `key` change is the refetch mechanism** — a changed `params` object alone does NOT refetch.
- The three list routes (`/api/photos`, `/api/albums/[id]/photos`, `/api/search`) all forward `parsed.data` wholesale to their service, so adding `sort` to the shared Zod schemas needs **no route changes**.
- Test runner is Vitest. Run one web file with `pnpm --filter @lumio/web exec vitest run <path>`; one shared file with `pnpm --filter @lumio/shared exec vitest run <path>`. `@lumio/web` resolves the `@/` alias in tests.
- Trash (`listTrash`, `trash-view.tsx`) is intentionally untouched — it keeps `[{ deletedAt: "desc" }, { id: "desc" }]` and gets no sort control. It reuses `photosQuerySchema`; an unused optional `sort` field there is harmless.

import type { PhotoSort } from "@lumio/shared";

/**
 * Map a sort choice to a Prisma `orderBy`: the chosen date field plus an `id`
 * tiebreaker in the SAME direction, so keyset/cursor pagination stays monotonic.
 * `sortDate` is `takenAt ?? importTime` (set at ingest), so the taken-date sorts
 * keep EXIF-less photos chronological by their import time. Shared by the
 * library/album/search listings and the detail-view neighbor query so they
 * paginate over the same sequence.
 */
export function photoOrderBy(sort?: PhotoSort) {
  switch (sort) {
    case "taken-asc":
      return [{ sortDate: "asc" as const }, { id: "asc" as const }];
    case "imported-desc":
      return [{ createdAt: "desc" as const }, { id: "desc" as const }];
    case "imported-asc":
      return [{ createdAt: "asc" as const }, { id: "asc" as const }];
    default: // "taken-desc"
      return [{ sortDate: "desc" as const }, { id: "desc" as const }];
  }
}

/** The default order (newest taken-date first). Used by the album-cover query,
 *  which always shows the most-recent representative regardless of user sort. */
export const PHOTO_ORDER = photoOrderBy();
```

import { DEFAULT_PHOTO_SORT, type PhotoSort } from "@lumio/shared";

/**
 * URL for a photo's detail route, carrying the navigation scope. An album id
 * scopes neighbors/film-strip to that album (else the whole library); a sort
 * (when not the default) tells the detail view which ordering to walk. This is
 * the one place the `?album=` / `?sort=` convention is defined.
 */
export function photoHref(id: string, albumId?: string | null, sort?: PhotoSort): string {
  const params = new URLSearchParams();

"use client";

import { useCallback, useSyncExternalStore } from "react";
import { coercePhotoSort, DEFAULT_PHOTO_SORT, type PhotoSort } from "@lumio/shared";

const STORAGE_KEY = "lumio:grid-sort";

/** Resolve the stored grid sort, defaulting to taken-date newest for
 *  missing/invalid input. Pure for testability. */
export function parseGridSort(stored: string | null): PhotoSort {
  return coercePhotoSort(stored ?? undefined);
}

// Same-document subscribers. The native `storage` event only fires in *other*
// tabs, so we keep our own listener set and notify it after a local write to keep
// grids in the current tab in sync.
const listeners = new Set<() => void>();

function subscribe(onChange: () => void) {
  listeners.add(onChange);
  window.addEventListener("storage", onChange);
  return () => {
    listeners.delete(onChange);
    window.removeEventListener("storage", onChange);
  };
}

function getSnapshot(): PhotoSort {
  return parseGridSort(localStorage.getItem(STORAGE_KEY));
}

// The server (and the first hydration pass) assume the default; the real value is
// read on the client after mount. useSyncExternalStore swaps to the client
// snapshot without a hydration mismatch.
function getServerSnapshot(): PhotoSort {
  return DEFAULT_PHOTO_SORT;
}

/**
 * Global, persisted photo sort order. Persisted to localStorage so the choice
 * carries across routes and reloads, and synced across tabs via the `storage`
 * event and across grids in the same tab via the local listener set.
 */
export function useGridSort() {
  const sort = useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot);

  const setSort = useCallback((next: PhotoSort) => {
    localStorage.setItem(STORAGE_KEY, next);
    listeners.forEach((cb) => cb());
  }, []);

  return { sort, setSort };
}
```

## Notes for the implementer

- **Refinement vs. spec:** the spec sketched `photoSortSchema = z.enum([...]).default(...)`. This plan instead exports `PHOTO_SORTS` + `DEFAULT_PHOTO_SORT` + a lenient `coercePhotoSort`, and makes the query-schema field `photoSortSchema.optional()`. Behaviour is identical (default = `taken-desc`), but the optional field keeps the existing service tests (which pass partial param objects) compiling, and `coercePhotoSort` is reused by both the hook and the detail-scope parser (DRY). Invalid `sort` on a list API request yields a 400, consistent with how the schemas already reject a bad `limit`; the detail route stays lenient via `coercePhotoSort`.
- **Why a `key` and not reactive `params`:** `usePhotoPages` fetches only on mount, so changing the sort must remount the grid. Including `sort` in the grid `key` is the established pattern (album/search already key on scope/filters).
- **Album covers are intentionally not sortable:** `listAlbumSummaries` keeps `PHOTO_ORDER` / `sortDate desc` so a thumbnail always shows the most-recent representative.
